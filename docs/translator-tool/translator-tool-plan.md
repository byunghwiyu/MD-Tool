# Translator Tool Plan

## 목표
로컬에서 실행되는 번역 툴을 만든다. 브라우저 UI에서 텍스트를 입력하거나 파일을 업로드하면 무료 번역 엔진을 통해 번역하고 결과를 저장할 수 있어야 한다.

## 제품 정의
- 로컬 전용 툴
- 브라우저 UI 사용
- `localhost`만 사용
- 무료 번역 엔진 또는 무료 플랜 API 사용
- 결과 미리보기와 저장 지원

## 핵심 사용자 시나리오
1. 사용자가 텍스트를 붙여넣거나 TXT/MD 파일을 업로드한다.
2. 원문 언어와 목표 언어를 선택한다.
3. 번역 엔진을 선택한다.
4. 번역을 실행한다.
5. 진행 상태를 확인한다.
6. 결과를 미리보고 저장한다.

## MVP 범위
- 텍스트 직접 입력 번역
- TXT/MD 파일 업로드 번역
- 언어 선택
- 번역 엔진 선택
- 진행 상태 표시
- 번역 결과 미리보기
- 결과 TXT/MD 저장
- 작업 이력

## 제외 범위
- 외부 공개 배포
- 로그인/회원 관리
- 협업 기능
- OCR 기반 PDF 번역
- 고급 용어집
- 번역 메모리 동기화

## 엔진 전략
여러 엔진을 교체 가능한 구조로 설계한다.

### 1순위
- LibreTranslate 계열 API
- 장점: 구조 단순, 무료 옵션 존재
- 단점: 공개 서버 품질 편차 가능

### 2순위
- Google Translate 비공식 라이브러리
- 장점: 체감 번역 품질 양호
- 단점: 비공식이라 차단 가능성 있음

### 3순위
- DeepL Free API
- 장점: 품질 우수
- 단점: API 키 필요, 무료 한도 존재

## 권장 구현 방향
엔진 인터페이스를 통일한다.

예시 인터페이스:
- `translate_text(text, source_lang, target_lang, options)`
- `translate_segments(segments, source_lang, target_lang, options)`

이 구조를 쓰면 엔진 추가와 교체가 쉬워진다.

## 아키텍처
- 백엔드: FastAPI
- 프론트엔드: 정적 HTML/CSS/JS
- 작업 처리: 로컬 job runner
- 실행 환경: localhost only

## 백엔드 책임
- 입력 텍스트/파일 수신
- 텍스트 분할 처리
- 엔진별 번역 호출
- 작업 상태 저장
- 결과 파일 생성
- 에러/재시도 처리

## 프론트엔드 책임
- 텍스트 입력 UI
- 파일 업로드 UI
- 언어 선택 UI
- 엔진 선택 UI
- 진행 상태 표시
- 결과 비교 및 다운로드 UI

## 처리 흐름
1. 요청 생성
2. 텍스트 정규화
3. 긴 텍스트를 세그먼트로 분리
4. 번역 엔진 호출
5. 세그먼트 병합
6. 결과 저장
7. UI 갱신

## 리스크
- 무료 API rate limit
- 비공식 API 차단
- 긴 문서 번역 속도 저하
- 문단 분리 품질 문제
- HTML/Markdown 포맷 손상 가능성

## 대응 전략
- 엔진 fallback 지원
- 세그먼트 단위 재시도
- 요청 속도 제한
- 작업 로그 저장
- 원문 보존 후 번역문 별도 생성

## 개발 단계
### 1단계
- 프로젝트 골격 생성
- localhost 서버 기동
- 기본 UI

### 2단계
- LibreTranslate 연동
- 텍스트 번역 MVP
- 결과 출력

### 3단계
- TXT/MD 파일 지원
- 세그먼트 처리
- 결과 저장

### 4단계
- Google 비공식 엔진 추가
- 엔진 선택 기능
- 오류 처리 개선

### 5단계
- 진행률/ETA 표시
- 작업 이력
- 결과 비교 뷰

### 6단계
- DeepL Free 추가
- 고급 옵션
- 포맷 보존 개선

## 성공 기준
- 텍스트 입력 번역 가능
- TXT/MD 번역 가능
- 엔진 2개 이상 선택 가능
- 실패 시 오류 표시
- 결과 저장 가능
- 로컬에서 안정적으로 반복 사용 가능
